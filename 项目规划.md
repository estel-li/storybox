你是一个资深全栈工程师，请帮我从零构建一个局域网儿童故事播放系统 MVP：老李讲故事。

项目采用 monorepo 单仓库结构，后端和 Web 前端放在同一个项目内不同目录中。

本次 MVP 只做：

* Go 后端
* GORM
* SQLite
* Nuxt 4 Web 端
* Nuxt UI 生态
* 本地运行
* 不做 Docker
* 不做 Android
* 不做 iOS
* 不做转码
* 不做复杂账号系统

核心目标是先跑通：

```text
扫描 NAS 故事目录 → 写入 SQLite → Web 后台查看和管理 → Web 端播放音频
```

# 一、项目背景

这是一个家庭局域网儿童故事播放系统。

NAS 是 Debian 系统，故事音频已经整理好。

当前故事库目录为：

```text
/volume3/16T/儿童故事/01_library_故事库
```

后端需要递归扫描这个目录，并根据目录结构自动识别分类、专辑、章节和故事。

Web 端用于：

* 家长后台管理
* 查看分类、专辑、故事
* 手动触发扫描
* 搜索故事
* 播放测试
* 简单儿童播放页面

第一版先不做 Android App，但后端 API 要为后续 Android App 预留。

# 五、故事库扫描规则

后端扫描根目录：

```text
STORY_LIBRARY_ROOT
```

实际示例：

```text
/volume3/16T/儿童故事/01_library_故事库
```

目录结构规则：

```text
一级目录 = 分类
二级目录 = 专辑
二级目录以下所有音频文件 = 故事
三级及更深目录 = 章节/分卷路径
```

示例：

```text
01_library_故事库/01_经典名著/三国演义（完结）/001群雄逐鹿/001.英雄问世三结义1.mp3
```

应识别为：

```text
分类：01_经典名著
专辑：三国演义（完结）
章节：001群雄逐鹿
故事标题：英雄问世三结义1
排序：001
```

支持音频格式：

```text
mp3
m4a
aac
wav
flac
ogg
```

忽略：

```text
jpg
jpeg
png
pdf
doc
docx
xls
xlsx
mp4
avi
mkv
flv
txt
sh
zip
```

排序解析规则：

支持这些文件名：

```text
001.标题.mp3
001 - 标题.mp3
001 标题.mp3
001_标题.mp3
01.标题.mp3
第001集 标题.mp3
000.引子.mp3
```

解析要求：

* 尽量提取前缀数字作为 `sort_order`
* 去掉前缀数字、分隔符、扩展名后作为 `title`
* 如果没有数字前缀，`sort_order = 999999`
* 同一专辑下按照 `chapter + sort_order + filename` 排序

# 六、数据库设计

字段：


说明：

* `name` 为目录原始名，例如 `01_经典名著`
* `display_name` 默认为去掉前缀数字后的名称，例如 `经典名著`
* `sort_order` 从目录前缀数字解析，无法解析则 999999

## albums 表


说明：

* `name` 为专辑目录原始名
* `display_name` 默认等于 `name`
* 第一版不强制做封面，`cover_path` 可为空

## stories 表

字段：

```text
id
category_id
album_id
chapter

display_title
sort_order
file_path
relative_path
file_ext
file_size
duration_seconds
file_hash

updated_at
```

说明：

* `file_path` 存真实路径，仅后端内部使用
* API 不要直接返回完整真实路径给普通播放端
* `relative_path` 可以返回，用于后台查看
* `duration_seconds` MVP 可以先为空或 0，不强制读取音频时长
* `file_hash` MVP 可选，第一版可以用 `relative_path + file_size + modified_time` 判断变化

# 十、后端实现细节要求

## 1. GORM 初始化

* 自动创建 SQLite 文件目录
* AutoMigrate 所有模型
* 建立必要索引

建议索引：

```text
categories.name
albums.category_id
albums.name
stories.album_id
stories.relative_path
stories.title
stories.is_visible
stories.is_missing
play_history.device_id
favorites.device_id
favorites.story_id
```

## 2. 扫描去重策略

优先使用 `relative_path` 作为唯一键。

重复扫描时：

* 如果 relative_path 已存在，更新文件大小、扩展名、标题、章节等
* 如果数据库记录存在但本次扫描没有发现，标记 `is_missing = true`
* 如果再次发现，标记 `is_missing = false`

## 3. 路径安全

播放接口必须根据 story id 查数据库里的 file_path。

不要允许用户通过 query 参数传任意路径播放文件。

## 4. CORS

允许：

```text
http://localhost:3000
```

也允许通过环境变量配置多个 origin。

## 5. 日志

使用 slog 或 zap，关键操作写日志：

* 服务启动
* 数据库初始化
* 扫描开始
* 扫描完成
* 扫描错误
* 播放文件不存在
* 登录失败
